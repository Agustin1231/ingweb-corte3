Fase 2 Entregable

Scrum + Git · Gestión del desarrollo

Conceptos: Scrum · Sprints · Historias de usuario · Git · Branching · Versión 1.0 · Última revisión 2026-05-05

1. Propósito

Este documento describe cómo se gestionó el desarrollo del sistema: la metodología ágil aplicada, el flujo de trabajo del equipo, la estrategia de ramas en Git y el detalle de cada uno de los tres sprints ejecutados.

Scrum no es solo "trabajar por sprints". Es una disciplina que vuelve visible el progreso, predecible la entrega y adaptable el alcance.

El equipo trabajó con sprints de 2 semanas, totalizando 6 semanas de desarrollo.

5. Definition of Ready y Definition of Done

5.1 Definition of Ready (DoR)

Una historia está lista para entrar a un sprint cuando:

  • Tiene narrativa "Como X, quiero Y, para Z".
  • Tiene criterios de aceptación verificables.
  • Está estimada en story points por el equipo.
  • Las dependencias técnicas están identificadas.
  • El diseño UX (si aplica) está acordado.

5.2 Definition of Done (DoD)

Una historia está terminada cuando se cumplen todos los siguientes criterios:

  • El código está en main tras una pull request revisada por al menos 1 par.
  • Las pruebas funcionales pasan localmente y en CI.
  • Los criterios de aceptación de la historia están verificados.
  • La documentación afectada está actualizada.
  • El sitio se desplegó correctamente en GitLab Pages.
  • No hay regresiones detectadas en historias previas.

7. Estrategia Git

Se adoptó un flujo simple basado en trunk-based con ramas cortas por historia, optimizado para CI/CD continuo a GitLab Pages.

Figura 2. Estrategia Git — ramas feat/* por historia, merge a main, tags por sprint.

7.1 Reglas del flujo Git

  • main es siempre desplegable. Cada merge dispara CI/CD a GitLab Pages.
  • Ramas cortas con prefijo feat/, fix/, docs/, chore/.
  • Pull requests con al menos 1 revisor antes de mergear.
  • Commits convencionales (Conventional Commits): tipo(scope): mensaje.
  • Tags semánticos al final de cada sprint: v0.1, v0.2, v0.3.
  • Sin force-push a main. Las ramas feature pueden rebase mientras no estén compartidas.

8. Sprint 1 · Autenticación

Sprint 1

📅 25 mar — 7 abr 2026 ⏱ 2 semanas 🎯 12 SP
Sprint Goal Permitir que un paciente cree una cuenta e inicie sesión de forma segura.
4Historias
12Story Points
2Endpoints
100%Completadas
HU-01 3 SP Hecho

Registro de paciente

Como paciente nuevo, quiero crear una cuenta con mi email y una contraseña, para poder usar el sistema y agendar citas.

Criterios de aceptación

  • El formulario valida email con formato correcto.
  • La contraseña debe tener al menos 8 caracteres con número y mayúscula.
  • Si el email ya existe, devuelve 409 EMAIL_DUPLICADO.
  • La respuesta exitosa es 201 Created con el perfil sin la contraseña.
HU-02 3 SP Hecho

Inicio de sesión

Como usuario registrado, quiero iniciar sesión con mis credenciales, para acceder a las funciones del sistema.

Criterios de aceptación

  • Al autenticar correctamente, devuelve un token Bearer y el perfil.
  • Credenciales inválidas devuelven mensaje genérico (no diferencia email vs password).
  • El token se almacena en el cliente para usarse en requests siguientes.
HU-03 3 SP Hecho

Cifrado de contraseñas

Como sistema, debo almacenar las contraseñas cifradas con hash + salt, para proteger las credenciales ante una eventual fuga de datos.

Criterios de aceptación

  • Ninguna contraseña se almacena en texto plano.
  • Cada usuario tiene un salt único.
  • Las pruebas verifican que el campo password_hash jamás sale en respuestas API.
HU-04 3 SP Hecho

Consulta de perfil propio

Como usuario autenticado, quiero ver mi perfil, para confirmar que mis datos están correctos.

Criterios de aceptación

  • GET /usuarios/me devuelve el perfil del titular del token.
  • Sin token devuelve 401.

HU-052 SPHecho

Listado de médicos

Como paciente, quiero ver la lista de médicos disponibles por especialidad, para escoger con quién agendar.

Criterios de aceptación

  • GET /medicos devuelve solo médicos activos.
  • Soporta filtro ?especialidad=.
HU-063 SPHecho

Consulta de disponibilidad

Como paciente, quiero consultar los horarios libres de un médico para una fecha, para escoger un slot disponible.

Criterios de aceptación

  • La respuesta lista solo slots no ocupados.
  • Fechas pasadas devuelven 422 FECHA_PASADA.
HU-075 SPHecho

Agendar cita

Como paciente, quiero agendar una cita en un horario disponible, para asegurar mi atención.

Criterios de aceptación

  • La validación de doble booking se hace de forma atómica en el servidor.
  • Si el slot fue tomado en paralelo, devuelve 409 CITA_DUPLICADA.
  • Confirmación visual al usuario con el detalle de la cita creada.
HU-083 SPHecho

Mis citas

Como paciente, quiero ver mis citas agendadas, para llevar control de mis próximas atenciones.

Criterios de aceptación

  • Solo se ven las citas del usuario autenticado.
  • Soporta filtro por estado (agendada, cancelada, atendida).
HU-095 SPHecho

Cancelar cita

Como paciente, quiero cancelar una cita propia, para liberar el horario y avisar a la clínica.

Criterios de aceptación

  • Solo el dueño puede cancelar su cita.
  • Cancelación bloqueada con menos de 1 h de anticipación.
  • El registro se conserva con estado cancelada (auditoría).

Vista global de citas

Como staff, quiero ver todas las citas con filtros (médico, estado, rango de fechas), para gestionar la operación diaria.

Criterios de aceptación

  • Listado paginado con filtros combinables.

Alta de médicos

Como admin, quiero registrar nuevos médicos con su especialidad, para mantener actualizado el catálogo.

Criterios de aceptación

  • Campos obligatorios: nombre, especialidad.
  • Solo rol admin puede crear médicos.
  • La acción se registra en el log de auditoría.
HU-123 SPHecho

Definición de horarios

Como admin/staff, quiero definir el horario semanal de atención de cada médico, para configurar la disponibilidad real.

Criterios de aceptación

  • Se pueden registrar múltiples bloques por día.
  • Validación: hora_inicio < hora_fin.
HU-135 SPHecho

Reportes operativos

Como admin, quiero ver métricas mensuales (ocupación, cancelaciones, citas por especialidad), para tomar decisiones operativas.

Criterios de aceptación

  • Reporte segmentado por rango de fechas.
  • Incluye ocupación promedio y desglose por especialidad.
HU-143 SPHecho

Auditoría de acciones administrativas

Como sistema, debo registrar las operaciones críticas (alta/baja de médicos, cancelaciones administrativas), para garantizar trazabilidad.

Criterios de aceptación

  • Cada acción crítica produce un registro con: usuario, acción, recurso, timestamp.
  • Los registros son inmutables.

12.1 Aprendizajes (resumen de retrospectivas)

  • Funcionó: commits convencionales y PRs pequeñas → revisiones rápidas y trazabilidad clara.
  • Funcionó: definir el contrato de API en Sprint 1 evitó retrabajo en Sprint 2 y 3.

  • A mejorar: automatizar más pruebas en CI para reducir verificación manual.
