Fase 1 Entregable

Arquitectura del sistema

Conceptos: Cliente-Servidor · Backend · Modularidad · API · W3C · Security by Design · Versión 1.0 · Última revisión 2026-05-05

1. Propósito del documento

Este documento describe la arquitectura de software del Sistema Web de Gestión de Citas Médicas. Su objetivo es:

  • Justificar el estilo arquitectónico elegido y sus alternativas.
  • Definir las vistas que permiten razonar sobre el sistema (lógica, procesos, despliegue, datos).
  • Establecer la modularización y los contratos entre módulos.
  • Documentar los principios de diseño y atributos de calidad.
  • Servir como fuente de verdad para el resto de fases.
La arquitectura precede al código. Cada decisión aquí condiciona qué tan mantenible, segura y escalable será la solución final.

2. Contexto del sistema

2.1 Problema

La clínica gestiona sus citas de forma manual, lo que produce:

  • Errores humanos (doble booking, datos mal escritos).
  • Demoras en atención telefónica y presencial.
  • Pérdida de información por uso de papel y registros descentralizados.
  • Dificultad de auditoría y reporte gerencial.

2.2 Solución

Una aplicación web que permita:

  • Pacientes: registrarse, consultar disponibilidad, agendar y cancelar citas.
  • Personal administrativo: gestionar médicos, horarios, agenda y reportes.
  • Administradores: control total del sistema y de los usuarios.

3. Estilo arquitectónico

3.1 Decisión: cliente-servidor en 3 capas

Figura 2. Estilo cliente-servidor en 3 capas — separación de responsabilidades.

3.2 ¿Por qué este estilo?

NecesidadCómo la atiende
Múltiples usuarios concurrentesEl servidor centraliza estado y reglas.
Acceso desde distintos dispositivosCliente web → estándares W3C.
Reglas de negocio críticasConcentradas en lógica, no replicadas en cliente.
Confidencialidad de datosLa capa de datos no queda expuesta al cliente.
MantenibilidadCada capa puede cambiar sin alterar las otras.

3.3 Alternativas consideradas y descartadas

AlternativaMotivo del descarte
Aplicación de escritorioNo es web; no accesible desde cualquier dispositivo.
Microservicios / serverlessSobre-ingeniería para el alcance del caso.
Cliente "gordo"Compromete seguridad e integridad.
Peer-to-peerNo aplica a un dominio centralizado como una clínica.

5. Modularización

5.1 Criterios aplicados

  • Alta cohesión — cada módulo agrupa funcionalidad de un mismo dominio.
  • Bajo acoplamiento — los módulos se comunican vía contratos.
  • Responsabilidad única — una sola razón de cambio por módulo.

5.2 Módulo de Usuarios

Responsabilidad: registrar, autenticar y gestionar perfiles y roles.

EndpointFunción
POST /auth/registerCrear cuenta de paciente.
POST /auth/loginIniciar sesión, devolver token.
GET /usuarios/mePerfil del usuario autenticado.

5.3 Módulo de Citas

Responsabilidad: ciclo de vida completo de una cita.

EndpointFunción
GET /disponibilidadHorarios libres por médico/fecha.
POST /citasAgendar cita.
GET /citasListar citas del usuario.
DELETE /citas/:idCancelar cita propia.

5.4 Módulo Administrativo

Responsabilidad: operación interna de la clínica.

EndpointFunción
GET /admin/citasTodas las citas con filtros.
POST /admin/medicosAlta de médicos.
POST /admin/horariosDefinir horarios.
GET /admin/reportesMétricas operativas.

5.5 Servicios transversales

ServicioFunción
AutenticaciónEmisión y validación de tokens.
AutorizaciónVerificación de rol antes de cada acción.
ValidaciónReglas de formato y negocio sobre los inputs.
Logging / AuditoríaRegistro de operaciones críticas.
Manejo de erroresRespuestas uniformes y mensajes seguros.

6. Comunicación entre componentes

  • Protocolo: HTTPS (TLS 1.2+).
  • Formato: JSON.
  • Estilo: REST.
  • Autenticación: token Bearer en header Authorization.

Códigos HTTP estándar

CódigoSignificado
200 OKOperación exitosa.
201 CreatedRecurso creado.
400 Bad RequestDatos inválidos.
401 UnauthorizedFalta autenticación.
403 ForbiddenSin permisos.
404 Not FoundRecurso inexistente.
409 ConflictConflicto (ej. doble booking).
500 Internal ErrorError no controlado.

    "detalles": {}

7. Principios de diseño aplicados

PrincipioAplicación
Separación de responsabilidades (SoC)Tres capas claramente delimitadas.
Bajo acoplamiento, alta cohesiónMódulos por dominio y comunicación por contrato.
ModularidadMódulos independientes, sustituibles y testeables.
Security by DesignAuth, validación y errores definidos desde el diseño.
Menor privilegioRoles diferenciados; cada uno solo accede a lo necesario.
Defensa en profundidadValidación tanto en cliente (UX) como en servidor.
Estandarización (W3C)HTML semántico y accesibilidad.
IdempotenciaLecturas sin efectos; escrituras protegidas contra duplicación.

9. Decisiones arquitectónicas (ADRs)

ADR-001 · Estilo cliente-servidor en 3 capas Aceptada

Contexto: Sistema multiusuario con reglas críticas y datos sensibles.

Decisión: Cliente-servidor con presentación, lógica y datos.

Consecuencias: Mayor mantenibilidad y seguridad; requiere disciplina para no mezclar responsabilidades.

ADR-002 · Comunicación REST + JSON Aceptada

Contexto: Necesidad de un contrato simple, ampliamente soportado y testeable.

Decisión: REST + JSON sobre HTTPS.

Consecuencias: Curva de aprendizaje baja, herramientas estándar, sencillo de versionar.

ADR-003 · Despliegue en GitLab Pages Aceptada

Contexto: Despliegue obligatorio en la nube; tiempo limitado.

Decisión: Frontend desde GitLab Pages mediante pipeline CI/CD.

Consecuencias: HTTPS gratuito, despliegue continuo; backend, de existir, requerirá servicio externo.

ADR-004 · Autenticación basada en token Aceptada

Contexto: Identificar al usuario en cada request sin sesiones del lado del servidor.

Decisión: Token Bearer emitido al login y validado en cada request.

Consecuencias: Backend sin estado; rotación y expiración deben gestionarse.

ADR-005 · Hash de contraseñas con salt Aceptada

Contexto: OWASP A02:2021 — Cryptographic Failures.

Decisión: Las contraseñas se almacenan con hash unidireccional + salt único.

Consecuencias: No se pueden recuperar contraseñas, solo restablecer; protección frente a fuga.

12. Riesgos arquitectónicos identificados

RiesgoOrigenMitigación
Doble booking Concurrencia en módulo de Citas. Restricción única (medico_id, fecha_hora).
Acceso indebido a datos Falla de autorización entre módulos. Validación de rol en cada endpoint.
Fuga de credenciales Almacenamiento incorrecto de contraseñas. Hash + salt obligatorio.
Indisponibilidad Punto único de fallo en backend. Frontend en CDN sigue sirviendo modo lectura.
