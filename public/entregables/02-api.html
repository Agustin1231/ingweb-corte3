Fase 2 Entregable

Contrato de la API

Conceptos: API · REST · JSON · HTTPS · Autenticación · Versionado · Versión 1.0 · Última revisión 2026-05-05

1. Propósito

Este documento define el contrato técnico entre el frontend y el backend del sistema. Establece qué recursos expone la API, cómo se invocan, qué datos intercambian y qué respuestas devuelven. Es la traducción operativa de la arquitectura.

Una API bien especificada permite que frontend y backend evolucionen en paralelo sin romperse mutuamente, y habilita pruebas automatizadas desde el inicio.

2. Principios y convenciones

PrincipioAplicación concreta
Estilo arquitectónicoREST — recursos identificables por URL, verbos HTTP semánticos.
Formato de intercambioJSON con codificación UTF-8.
TransporteHTTPS obligatorio (TLS 1.2+).
Sin estadoCada request es independiente; el estado vive en el cliente (token).
IdempotenciaGET, PUT y DELETE son idempotentes; POST crea recursos.
VersionadoPrefijo de versión en la URL: /api/v1/... (ver §11).
PaginaciónListados grandes usan ?page=&limit= con metadatos de paginación.
IdentificadoresUUID o entero positivo; nunca exponer información sensible en el ID.
FechasISO 8601 en UTC (2026-05-15T14:30:00Z).
NamingRecursos en plural y minúscula (/citas, /medicos).

2.1 Headers comunes

HeaderUso
Content-Type: application/jsonEn todo request con body.
Accept: application/jsonRecomendado en todo request.
Authorization: Bearer <token>En endpoints autenticados.

3. Autenticación

La API usa autenticación basada en token Bearer emitido al iniciar sesión. El token debe enviarse en cada request a endpoints protegidos mediante el header Authorization.

Figura 1. Flujo de autenticación con token Bearer.

3.1 Roles del sistema

RolPermisos
pacienteGestionar sus propias citas y perfil.
staffVer y gestionar citas de la clínica, registrar médicos y horarios.
adminAcceso total + gestión de usuarios y reportes.

4. Errores estándar

4.1 Códigos HTTP

CódigoSignificadoCuándo se usa
200 OKÉxito con cuerpoGET, DELETE con respuesta.
201 CreatedRecurso creadoPOST exitoso.
204 No ContentÉxito sin cuerpoDELETE sin retorno.
400 Bad RequestDatos inválidosValidación de schema fallida.
401 UnauthorizedFalta token o inválidoEndpoint protegido sin auth.
403 ForbiddenSin permisosAutenticado pero rol insuficiente.
404 Not FoundRecurso inexistenteID no encontrado.
409 ConflictConflicto de estadoDoble booking, email duplicado.
422 UnprocessableReglas de negocioFecha pasada, formato válido pero inválido contextualmente.
500 Server ErrorError no controladoBug; nunca debe filtrar internals.

4.2 Esquema uniforme de error

Todas las respuestas de error siguen exactamente esta estructura:

{
  "error": {
    "codigo": "CITA_DUPLICADA",
    "mensaje": "Ya existe una cita en ese horario para el médico seleccionado",
    "detalles": {
      "medico_id": 12,
      "fecha_hora": "2026-05-15T14:30:00Z"
    }
  }
}
CampoDescripción
codigoIdentificador estable, mayúsculas con underscore. Permite manejo programático.
mensajeTexto humano en español, seguro para mostrar al usuario.
detallesObjeto opcional con contexto adicional. Nunca incluye stacktraces ni datos sensibles.

6. Endpoints · Autenticación y Usuarios

POST /api/v1/auth/register Público

Crea una cuenta de paciente. Realiza hash de la contraseña con salt antes de persistir.

Request body
{
  "nombre": "Ana Pérez",
  "email": "ana@example.com",
  "password": "MiClav3Segura!"
}
Response 201
{
  "id": "u_8c1d3f",
  "nombre": "Ana Pérez",
  "email": "ana@example.com",
  "rol": "paciente",
  "creado_en": "2026-05-05T18:42:00Z"
}
Errores
CódigoCausa
400 EMAIL_INVALIDOFormato de email no válido.
400 PASSWORD_DEBILNo cumple política mínima.
409 EMAIL_DUPLICADOYa existe una cuenta con ese email.
POST /api/v1/auth/login Público

Autentica al usuario y devuelve un token Bearer.

Request body
{
  "email": "ana@example.com",
  "password": "MiClav3Segura!"
}
Response 200
{
  "token": "eyJhbGciOi...",
  "expiresIn": 3600,
  "user": {
    "id": "u_8c1d3f",
    "nombre": "Ana Pérez",
    "rol": "paciente"
  }
}
Errores
CódigoCausa
401 CREDENCIALES_INVALIDASMensaje genérico — no diferencia "email no existe" de "password incorrecto" para evitar enumeración.
429 DEMASIADOS_INTENTOSRate limit por IP/email superado.
GET /api/v1/usuarios/me Requiere token

Devuelve el perfil del usuario autenticado.

Headers
Authorization: Bearer <token>
Response 200
{
  "id": "u_8c1d3f",
  "nombre": "Ana Pérez",
  "email": "ana@example.com",
  "rol": "paciente",
  "creado_en": "2026-05-05T18:42:00Z"
}

7. Endpoints · Disponibilidad y Médicos

GET /api/v1/disponibilidad Requiere token

Devuelve los horarios libres de un médico para una fecha dada.

Query params
?medico_id=m_42
&fecha=2026-05-15
Response 200
{
  "medico_id": "m_42",
  "fecha": "2026-05-15",
  "slots": [
    "08:00", "08:30", "09:00",
    "10:30", "11:00"
  ]
}
Errores
CódigoCausa
400 FECHA_INVALIDAFormato fecha incorrecto.
404 MEDICO_NO_ENCONTRADOEl médico no existe o está inactivo.
422 FECHA_PASADALa fecha consultada ya pasó.
GET /api/v1/medicos Requiere token

Lista los médicos activos, opcionalmente filtrados por especialidad.

Query params
?especialidad=cardiologia
&page=1&limit=20
Response 200
{
  "data": [
    { "id": "m_42", "nombre": "Dr. López",
      "especialidad": "cardiologia" }
  ],
  "page": 1, "limit": 20, "total": 8
}

8. Endpoints · Citas

POST /api/v1/citas Requiere token

Agenda una nueva cita para el usuario autenticado. Valida disponibilidad atómicamente.

Request body
{
  "medico_id": "m_42",
  "fecha_hora": "2026-05-15T09:00:00Z",
  "motivo": "Control anual"
}
Response 201
{
  "id": "c_a93f2b",
  "usuario_id": "u_8c1d3f",
  "medico_id": "m_42",
  "fecha_hora": "2026-05-15T09:00:00Z",
  "estado": "agendada",
  "creado_en": "2026-05-05T19:00:00Z"
}
Errores
CódigoCausa
409 CITA_DUPLICADAYa existe una cita en ese horario para el médico.
422 FECHA_PASADANo se puede agendar en el pasado.
422 FUERA_DE_HORARIOEl médico no atiende en ese horario.
404 MEDICO_NO_ENCONTRADOEl médico referenciado no existe.
GET /api/v1/citas Requiere token

Lista las citas del usuario autenticado, ordenadas por fecha descendente.

Query params
?estado=agendada
&desde=2026-05-01
&page=1&limit=10
Response 200
{
  "data": [
    { "id": "c_a93f2b",
      "medico": { "id":"m_42","nombre":"Dr. López" },
      "fecha_hora": "2026-05-15T09:00:00Z",
      "estado": "agendada" }
  ],
  "page": 1, "limit": 10, "total": 1
}
GET /api/v1/citas/:id Requiere token

Detalle de una cita propia. Devuelve 403 si no pertenece al usuario (excepto staff/admin).

Errores
CódigoCausa
403 NO_AUTORIZADOLa cita no pertenece al usuario.
404 CITA_NO_ENCONTRADAEl ID no existe.
DELETE /api/v1/citas/:id Requiere token

Cancela una cita propia. Cambia el estado a cancelada en lugar de eliminar físicamente (auditoría).

Response 200
{
  "id": "c_a93f2b",
  "estado": "cancelada",
  "cancelada_en": "2026-05-06T10:15:00Z"
}
Notas
  • Solo se puede cancelar con > 1h de anticipación.
  • El registro permanece para auditoría.
  • Solo el dueño o un staff/admin puede cancelar.

10. Modelos de datos (DTOs)

Resumen de los esquemas usados en requests y responses. Coincide con el modelo conceptual de la Fase 1.

10.1 Usuario

{
  "id": "string (uuid o slug)",
  "nombre": "string (1..120)",
  "email": "string (email, único)",
  "rol": "paciente | staff | admin",
  "creado_en": "string (ISO 8601 UTC)"
}
password_hash es interno, jamás se expone en respuestas.

10.2 Cita

{
  "id": "string",
  "usuario_id": "string",
  "medico_id": "string",
  "fecha_hora": "string (ISO 8601 UTC)",
  "estado": "agendada | confirmada | cancelada | atendida",
  "motivo": "string (opcional, 0..500)",
  "creado_en": "string",
  "cancelada_en": "string (opcional)"
}

10.3 Médico

{
  "id": "string",
  "nombre": "string",
  "especialidad": "string",
  "activo": "boolean"
}

10.4 Horario

{
  "id": "string",
  "medico_id": "string",
  "dia_semana": "lunes..domingo",
  "hora_inicio": "HH:mm",
  "hora_fin": "HH:mm"
}

10.5 Paginación (envoltorio común)

11. Versionado de la API

AspectoPolítica
Versión actualv1 (prefijo de URL /api/v1)
Cambios menoresAdicionales y compatibles hacia atrás → no incrementan versión.
Cambios mayoresCualquier ruptura de contrato → nueva versión v2 en paralelo.
DeprecaciónHeader Deprecation + plazo mínimo de 90 días antes de retirar.
DocumentaciónEste documento se versiona junto con el código en Git.

12. Notas de seguridad

Resumen de decisiones que se profundizan en Fase 5 — Seguridad:

  • HTTPS obligatorio en todos los entornos.
  • Hash + salt de contraseñas; jamás en texto plano ni reversible.
  • Validación autoritativa en servidor, incluso si el cliente ya validó.
  • Mensajes de error genéricos en login/registro para evitar enumeración de usuarios.
  • Rate limiting en endpoints sensibles (login, register).
  • Control de acceso por rol verificado en cada endpoint.
  • Sin filtración de internals en respuestas de error 500.
  • Auditoría de operaciones críticas (alta/baja de médicos, cancelaciones administrativas).
