Fase 3 Entregable

Despliegue en la nube

Conceptos: Nube · CI/CD · GitLab Pages · HTTPS · Versión 1.0 · Última revisión 2026-05-05

1. Propósito

Este documento describe cómo y dónde se publica el sistema para que sea accesible públicamente vía internet, qué pipeline lo automatiza, y qué mecanismos garantizan la disponibilidad y trazabilidad del despliegue.

"Funciona en mi máquina" no es una entrega. La nube es la única forma honesta de demostrar que el sistema realmente está vivo.

2. Modelo de despliegue elegido

Se adoptó una estrategia static-first sobre GitLab Pages: el frontend se publica como sitio estático servido desde un CDN, y el backend (cuando exista) corre en un servicio externo invocado vía API REST.

4. Archivo .gitlab-ci.yml

El pipeline se declara en un único archivo en la raíz del repositorio:

# ───────────────────────────────────────────────
# Pipeline de despliegue · Sistema de Citas Médicas
# ───────────────────────────────────────────────
stages:
  - validate
  - build
  - deploy

# 1. Validación de HTML y enlaces
validate:
  stage: validate
  image: node:20-alpine
  script:
    - npx --yes html-validate "public/**/*.html"
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

# 2. Construcción de artefactos
build:
  stage: build
  image: alpine:latest
  script:
    - echo "Copiando sitio estático..."
    - mkdir -p artifacts
    - cp -r public/* artifacts/
  artifacts:
    paths:
      - artifacts
    expire_in: 1 week
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

# 3. Publicación en GitLab Pages
pages:
  stage: deploy
  dependencies:
    - build
  script:
    - mv artifacts public
  artifacts:
    paths:
      - public
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
  environment:
    name: production
    url: https://$CI_PROJECT_NAMESPACE.gitlab.io/$CI_PROJECT_NAME

Decisiones clave del pipeline:

  • Tres stages (validate → build → deploy): si la validación falla, no se construye ni se publica.
  • Imagen mínima alpine:latest para los pasos no-Node — pipeline rápido.
  • rules por rama: solo main dispara el deploy; los MR solo validan.
  • environment: GitLab muestra la URL del despliegue en su UI con botón "View".
  • expire_in: artefactos intermedios se eliminan a la semana para no consumir cuota.

6. Dominio y HTTPS

  • HTTPS automático — GitLab provisiona certificado Let's Encrypt sin configuración adicional.
  • HSTS — habilitado por defecto en el dominio *.gitlab.io.
  • Dominio personalizado — opcional; requiere configurar registros DNS (CNAME) y verificación TXT. Para esta entrega se usa el dominio por defecto.

8. Estrategia de rollback

Si un despliegue introduce un problema crítico, hay dos vías de rollback:

  1. Revert por Git (preferido): git revert <hash> y push. Es trazable y queda en la historia. El pipeline se ejecuta y republica la versión anterior.
  2. Re-deploy de pipeline previo: en GitLab → CI/CD → Pipelines, se selecciona el último pipeline verde y se hace "Retry job pages". No deja registro en Git.
Regla: nunca usar git push --force a main. El historial es la auditoría del proyecto.

9. Monitoreo y observabilidad

Al ser un sitio estático, el monitoreo es liviano pero existe:

  • GitLab Pipelines — historial completo de despliegues con estado y duración.
  • GitLab Environments — vista de despliegues activos y URL asociada.
  • UptimeRobot / GitHub Status (opcional) — ping HTTP cada 5 minutos a la URL pública.
  • Lighthouse CI (futuro) — métricas de calidad por release.

11. Checklist de despliegue

  • ☑ Repositorio creado en GitLab.
  • ☑ Visibilidad establecida en "Público".
  • ☑ .gitlab-ci.yml agregado en la raíz.
  • ☑ Carpeta public/ con el sitio.
  • ☑ Pipeline ejecutado exitosamente al menos una vez.
  • ☑ URL pública verificada en navegador.
  • ☑ HTTPS activo (icono de candado en navegador).
  • ☑ Pruebas en al menos un dispositivo móvil real.
  • ☑ URL documentada en el README del proyecto.
