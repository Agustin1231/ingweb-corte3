Fase 4 Entregable

Calidad del software · ISO/IEC 25010

Conceptos: ISO 25010 · Testing · UX · Mantenibilidad · Versión 1.0 · Última revisión 2026-05-05

1. Propósito

Este documento evalúa el sistema contra el modelo de calidad ISO/IEC 25010, describe el plan de pruebas ejecutado y consolida las métricas obtenidas. La calidad no se declara: se mide.

2. Norma ISO/IEC 25010

La norma define ocho características de calidad del producto software, cada una descompuesta en sub-características. Es el marco de referencia internacional más utilizado para evaluar sistemas.

4. Las 8 características en detalle

4.1 Adecuación funcional 95

  • Completitud: los 14 requisitos del backlog están implementados.
  • Corrección: la salida del sistema coincide con lo esperado en todos los casos felices y de error.
  • Pertinencia: el sistema ofrece exactamente las funciones requeridas, sin features superfluas.

4.2 Eficiencia de desempeño 90

  • Tiempo de respuesta: sitio estático servido desde CDN — TTFB < 200 ms.
  • Uso de recursos: CSS y JS minimalistas, sin frameworks pesados.
  • Capacidad: escalabilidad implícita por estar en CDN global.

4.3 Compatibilidad 85

  • Coexistencia: no requiere instalación; convive con cualquier otra aplicación web.
  • Interoperabilidad: API REST + JSON estándar, fácilmente integrable.

4.4 Usabilidad 80

  • Reconocibilidad: portal con tarjetas claras y propósito explícito.
  • Aprendibilidad: flujo de agendamiento en menos de 3 pasos.
  • Operabilidad: formularios con validación inline.
  • Estética: diseño consistente con paleta unificada.
  • Accesibilidad: HTML semántico, etiquetas aria-label en navegación y SVGs.

4.5 Fiabilidad 75

  • Madurez: sin defectos críticos detectados en pruebas de aceptación.
  • Disponibilidad: heredada de GitLab Pages (SLA > 99.9%).
  • Tolerancia a fallos: manejo uniforme de errores con esquema único.
  • Recuperabilidad: rollback documentado vía Git revert.

4.6 Seguridad 85

Profundizada en la Fase 5 — Seguridad web.

  • HTTPS obligatorio, hash + salt de contraseñas, control de acceso por roles, validación autoritativa.

4.7 Mantenibilidad 90

  • Modularidad: módulos por dominio (usuarios, citas, admin) con responsabilidad única.
  • Reusabilidad: CSS compartido + componentes consistentes (cards, endpoint cards, story cards).
  • Analizabilidad: documentación exhaustiva en /docs/ y entregables HTML.
  • Modificabilidad: separación de capas permite cambiar UI sin tocar lógica.
  • Capacidad de prueba: contratos API explícitos facilitan pruebas automatizadas.

4.8 Portabilidad 95

  • Adaptabilidad: diseño responsive funciona en cualquier viewport.
  • Instalabilidad: es una URL — no requiere instalación.
  • Reemplazabilidad: cualquier hosting estático puede sustituir a GitLab Pages.

8. UX y accesibilidad

  • Navegación consistente: mismo header en todas las páginas, brand siempre visible.
  • Jerarquía visual clara: uso de headings, badges, callouts para guiar lectura.
  • Feedback inmediato: hover states en cards, focus visibles para teclado.
  • Etiquetado semántico: <header>, <main>, <article>, <aside>, <footer>.
  • Atributos ARIA: aria-label en navegación y todos los SVG diagramas.
  • Contraste: texto principal supera la ratio 7:1 (AAA) en la mayoría de bloques.
  • Responsive: breakpoint a 860 px colapsa el TOC lateral debajo del contenido.

9. Mantenibilidad del código

  • Estructura predecible: /public sirve, /docs documenta — separación clara.
  • CSS centralizado: un solo styles.css con variables CSS para tema unificado.
  • Sin duplicación: componentes reusados (cards, callouts, tags, endpoint cards).
  • Documentación viva: cada decisión arquitectónica está registrada como ADR.
  • Convenciones de nombres: archivos numerados NN-tema facilitan ordenamiento.
