Seguridad web

Conceptos: OWASP · Cifrado · Autenticación · Confidencialidad · Security by Design · Versión 1.0 · 2026-05-05

1. Propósito

Este documento describe las medidas de seguridad incorporadas al sistema, las verifica contra el referente OWASP Top 10 y demuestra que la seguridad es parte del diseño, no un añadido posterior.

La seguridad parchada después es cara y frágil. La seguridad diseñada desde el principio es invisible y robusta.

2. Security by Design

El proyecto adopta los principios fundamentales de Security by Design:

PrincipioAplicación
Menor privilegioRoles diferenciados (paciente, staff, admin); cada uno solo accede a lo necesario.
Defensa en profundidadMúltiples capas de control: HTTPS, auth, autorización, validación, hash.
Fail secureSi algo falla, el sistema cierra acceso por defecto (deny-by-default).
Mediación completaCada request pasa por validación de auth y rol; sin atajos.
Diseño abiertoLa seguridad no depende de "secretos del código"; depende de claves bien gestionadas.
Separación de privilegiosRoles independientes; el admin no es el dueño del recurso, es el supervisor.
Mínimo común denominadorValidación autoritativa en servidor, sin asumir nada del cliente.
Aceptabilidad psicológicaMensajes de error claros sin filtrar información sensible.

4. OWASP Top 10 — análisis

Estado del proyecto frente al OWASP Top 10 (edición 2021):

5. Autenticación y autorización

5.1 Autenticación

  • Método: token Bearer firmado, emitido al login.
  • Validez: 1 hora; el cliente debe re-autenticarse o usar refresh token.
  • Mensaje de error: genérico (CREDENCIALES_INVALIDAS) — no diferencia "email no existe" vs "password incorrecto".
  • Rate limiting: 5 intentos por IP/email cada 5 minutos.
  • Política de password: ≥ 8 caracteres, al menos 1 número y 1 mayúscula.

6. Cifrado

El sistema aplica cifrado en dos capas claramente diferenciadas:

CapaQué protegeMecanismo implementadoRecomendado en producción
En tránsito Toda comunicación navegador ↔ hosting HTTPS / TLS 1.3 con certificado gestionado por GitHub Pages. Igual + HSTS preload.
Contraseñas Credencial del usuario en almacenamiento Hash unidireccional SHA-256 + salt único por usuario, vía crypto.subtle (Web Crypto API). La contraseña original no se almacena nunca. bcrypt / scrypt / Argon2 (hashes lentos diseñados para passwords).
Tokens de sesión Identificador de sesión activa UUID v4 generado con crypto.randomUUID(), expira en 1 h. JWT firmado (HMAC/RSA) o token opaco con introspección server-side.
Datos personales Citas, perfiles Persistencia en localStorage del cliente (sin cifrar — limitación inherente del demo). Cifrado a nivel de BD (TDE) y aislamiento por tenant.
Nota sobre el alcance del demo. Esta entrega es una SPA estática con persistencia en cliente (localStorage), por lo que no existen credenciales de servidor que cifrar ni una BD que aislar. Lo que sí está implementado y verificable en el código es: (1) HTTPS en transporte vía GitHub Pages, y (2) hash SHA-256 con salt en contraseñas (ver app.js función sha256()). Las celdas "recomendado en producción" describen la evolución natural si el sistema migrara a un backend real.
Lo que NUNCA hacemos: almacenar contraseñas en texto plano, transmitir credenciales por GET, exponer hashes en respuestas de API, usar MD5 o SHA-1 para passwords.

7. Confidencialidad de datos personales

  • Principio de mínima exposición: los endpoints solo devuelven los campos necesarios.
  • Sin PII en logs: los logs no registran emails, contraseñas ni datos identificables.
  • Sin PII en URLs: los identificadores son opacos (UUID/slug), no valores sensibles.
  • Borrado lógico: las cancelaciones marcan estado, no eliminan; el usuario puede solicitar borrado total (derecho al olvido).
  • Política de retención: registros de auditoría se conservan 12 meses.

8. Validación e inputs

  • Doble validación: el cliente valida para UX, el servidor valida con autoridad.
  • Validación por schema: cada endpoint declara los tipos esperados; lo que no encaja se rechaza con 400.
  • Allow-list, no deny-list: se aceptan formatos conocidos en lugar de rechazar formatos malos.
  • Límites de tamaño: bodies con tope para prevenir DoS por payload masivo.
  • Escapado de salida: todo dato proveniente del usuario se escapa antes de renderizar (anti-XSS).

9. HTTPS y headers de seguridad

Headers que el servidor (o el meta-tag del cliente para el caso estático) declara:

Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'self'; img-src 'self' data:;
                          style-src 'self' 'unsafe-inline'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer-when-downgrade
Permissions-Policy: geolocation=(), camera=(), microphone=()

11. Plan de respuesta a incidentes

  1. Detección: revisar logs y métricas; alertas si hay patrones anómalos (picos de 401, accesos a /admin desde IPs no usuales).
  2. Contención: revocar tokens activos, deshabilitar cuentas comprometidas, rotar claves del servidor.
  3. Erradicación: identificar la causa raíz, desplegar parche, validar.
  4. Recuperación: restaurar desde backup si hubo pérdida; comunicar a usuarios afectados.
  5. Lecciones aprendidas: documentar el incidente, ajustar pruebas y monitoreo.
